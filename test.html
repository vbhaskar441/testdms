Deals Management Service (DMS) refers to the process of organizing, tracking, maintaining, prioritizing, and analyzing deals throughout their lifecycle to maximize value and efficiency. Here are the steps involved in implementing a DMS:

  1. Process Planning Stage:

    • Define the deal management process that suits your company's needs and resources.
    • Identify the types of deals, responsible parties, past issues, and necessary resources.

  2. Process Implementation Stage:

    • Deploy an automated deal management system.
    • Import data and onboard suppliers and team members to the new system.

  3. Pre-Deal Stage:

    • Negotiate trading programs with partners.
    • Provide commercial teams with necessary data for beneficial agreements.

  4. Deal Stage:

    • Execute the deal according to the agreed terms.
    • Monitor the deal's progress and performance.

  5. Handover Stage:

    • Transition the deal to the execution phase.
    • Ensure all stakeholders are informed and aligned.

  6. Pre-Renegotiation Stage:

    • Review the deal's performance and identify areas for improvement.
    • Prepare for renegotiation if necessary.

  7. Post-Deal Stage:

    • Analyze the deal's outcomes and document lessons learned.
    • Use insights to improve future deal management processes.

2. Scripting

GlideAjax

  • Definition: Use GlideAjax to fetch deal-related data from a third-party system.
  • Example:
    1. Script Include: 
      var DealDataFetcher = Class.create();
DealDataFetcher.prototype = {
    initialize: function() {},
    getDealData: function(dealId) {
        // Logic to fetch data from third-party system
        return "Deal data for " + dealId;
    },
    type: 'DealDataFetcher'
};
    2. Client Script: 
      function fetchDealData() {
    var ga = new GlideAjax('DealDataFetcher');
    ga.addParam('sysparm_name', 'getDealData');
    ga.addParam('dealId', g_form.getValue('deal_id'));
    ga.getXMLAnswer(function(response) {
        var answer = response.responseXML.documentElement.getAttribute("answer");
        g_form.setValue('deal_data', answer);
    });
}

Business Rule

  • Definition: A rule to update the deal status when a related task is completed.
  • Example: 
    (function executeRule(current, previous /*null when async*/) {
    if (current.state == 'Closed Complete') {
        var deal = new GlideRecord('u_deal');
        if (deal.get(current.u_deal)) {
            deal.setValue('status', 'Completed');
            deal.update();
        }
    }
})(current, previous);

3. Transform Map

Transform Map Scripts

  • Definition: Scripts to transform incoming deal data to the target table format.
  • Example:
    • onBefore Script: Executes before the source row is transformed into the target row. 
      (function transformRow(source, target, map, log, action) {
    // Example: Set default status if not provided
    if (!source.status) {
        target.status = 'New';
    }
})(source, target, map, log, action);
    • onAfter Script: Executes after the source row has been transformed into the target row. 
      (function transformRow(source, target, map, log, action) {
    // Example: Log the transformation
    log.info('Transformed deal: ' + target.number);
})(source, target, map, log, action);

1. Client Scripts

  • OnLoad: Executes when a form is loaded.

  • OnChange: Executes when a field value changes.

    function onChange(control, oldValue, newValue, isLoading) {
    if (newValue === 'High') {
        g_form.setMandatory('priority', true);
    }
}

  • OnSubmit: Executes when a form is submitted.

    function onSubmit() {
    if (g_form.getValue('short_description') === '') {
        alert('Short description is mandatory');
        return false;
    }
    return true;
}

  • OnCellEdit: Executes when a cell in a list is edited.

    function onCellEdit(sysIDs, table, oldValues, newValue, callback) {
    if (newValue === 'Closed') {
        alert('Cannot change state to Closed');
        callback(false);
    } else {
        callback(true);
    }
}

Core Tables

  1. Task [task]: The base table for all task-related tables like Incident, Change, Problem, etc.
  2. Configuration Item [cmdb_ci]: The base table for all configuration items in the CMDB.
  3. User [sys_user]: Stores user records.
  4. Group [sys_user_group]: Stores user group records.
  5. Role [sys_user_role]: Stores user role records.

Data Dictionary Tables

  1. Tables [sys_db_object]: Stores metadata about each table in the ServiceNow database.
  2. Dictionary [sys_dictionary]: Defines the fields for each table, including their data types, lengths, default values, and other attributes.
  3. Documentation [sys_documentation]: Stores the human-readable labels, help text, and language information for fields

IT Service Management (ITSM) Tables

  1. Incident [incident]: Stores incident records.
  2. Change Request [change_request]: Stores change request records.
  3. Problem [problem]: Stores problem records.
  4. Service Catalog [sc_cat_item]: Stores service catalog items.
  5. Request [sc_request]: Stores service catalog requests.
  6. Requested Item [sc_req_item]: Stores individual items requested in a service catalog request.
  7. Catalog Task [sc_task]: Stores tasks related to service catalog requests.

Configuration Management Database (CMDB) Tables

  1. CI Relationship [cmdb_rel_ci]: Stores relationships between configuration items.
  2. Hardware [cmdb_ci_hardware]: Stores hardware configuration items.
  3. Software [cmdb_ci_software]: Stores software configuration items.

Knowledge Management Tables

  1. Knowledge [kb_knowledge]: Stores knowledge articles.

Workflow and Automation Tables

Security Tables
